FINTECH · ACCESSIBILITY · WCAG 2.1 AA · CRYPTO TRADING

Making Crypto Trading Accessible to 4 Crore Indians

Redesigning Giottus Futures for color-vision-deficient traders — the first WCAG 2.1 AA-compliant futures interface in Indian fintech, delivered in 5 weeks.

RoleLead Product Designer
Timeline5 Weeks
FocusIA · WCAG · FinTech UX
ToolsFigma · FigJam · WCAG Audit

THE PROBLEM

I went in to fix a navigation problem. I discovered a market exclusion problem.

Giottus Futures is a high-frequency derivatives platform used by thousands of active traders daily. The brief was straightforward: traders were taking too long to place orders. Fix the navigation.

What the heatmap data revealed was deeper. Traders were switching between the Chart tab and Order Form tab an average of 4.2 times per trade — spending 102 seconds on a task that should take under 30. The fragmented 3-tab architecture was the root cause.

But while interviewing CVD traders, a second, far more severe problem emerged: 8% of Indian males — approximately 4 crore people — have some form of color vision deficiency. For them, the platform wasn't difficult. It was completely unusable. Buy and Sell buttons were green and red and nothing else. Price direction had no non-color indicator. Order status was invisible. This was a market exclusion problem hiding behind a navigation problem.

I went in to fix navigation. The fix inadvertently unlocked 4 crore users nobody else had reached.

TWO PROBLEMS, ONE REDESIGN

Accessibility failure. Architecture failure. Both at the same time.

My audit revealed two completely separate but equally severe problems that required an architectural solution — not a visual refresh.

  • Problem 1 — Accessibility failure: Buy/Sell buttons were green and red. Price movement used color alone. For a CVD user, the platform was completely unusable — not just difficult. Every critical action was invisible.
  • Problem 2 — Architecture failure: Even for non-CVD users, a heatmap study revealed traders spent an average of 102 seconds per order — over a minute of which was pure navigation overhead from a fragmented 3-tab interface. This was causing a 15% order abandonment rate.
Existing Interface

RESEARCH & DISCOVERY

I don't design until I understand the problem at a systemic level.

  • User interviews (6 traders): Recruited 6 traders — one with colour vision differences had abandoned the platform, not because it was hard to use, but because it was impossible to use safely without risking financial loss.
  • Heatmap & session recording analysis (200+ sessions): Found the 3-tab navigation was the root cause of the 102s task time. Users were switching between Chart and Order Form an average of 4.2 times per trade.
  • Competitor audit: Checked every major Indian crypto exchange — WazirX, CoinDCX, Zebpay. Not one had a CVD accessibility mode. This was a competitive moat, not just a compliance fix.
  • WCAG 2.1 AA audit: The platform failed on 18 WCAG AA criteria. Most critical: 1.4.1 (Use of Color), 1.4.3 (Contrast Ratio), and 1.4.11 (Non-text Contrast).

THE SOLUTION

One unified interface. Three color treatments. Zero compromises.

Architectural fix first: Collapsed the 3-tab navigation into a single unified trading view. Chart, order form, and active positions visible simultaneously — no more switching. Task time: 102s → 34s (67% reduction).

Then the unexpected: While validating the unified layout with CVD traders, it became clear that collapsing the interface had inadvertently solved the color-reliance problem too — every element now had spatial, positional, and shape context. This gave a foundation to build the full accessibility system on.

Accessibility system: A single Figma design system with three simultaneous color treatments — each achieving WCAG 2.1 AA + IS 17802:2021 compliance independently:

  • Default Mode: Redesigned with improved contrast ratios and redundant iconographic indicators alongside every color cue.
  • Colorblind Mode: Blue (Buy) and Yellow (Sell) — the most distinguishable pair across all CVD types. Every color-only indicator replaced with color + icon + label.
  • Greyscale Mode: True 9:1 contrast ratio for outdoor traders and severe CVD. Filled shapes (■) for Buy, outlined shapes (□) for Sell. Zero color reliance.

KEY DESIGN DECISIONS

Every decision had a reason. Here are the ones that mattered most.

  • Blue/Yellow for CVD mode, not red/green inversion: Blue-yellow is the most distinguishable pair across all CVD types. Red-green alternatives only work for one CVD type.
  • Redundant non-color indicators on every critical element: Every Buy/Sell action has: a color, an icon (arrow), a text label, AND a shape treatment. A user who perceives zero color can still trade safely.
  • Mode selector persistent across sessions: The preference is stored in localStorage. A CVD trader sets it once — then it's invisible infrastructure forever.
  • Single Figma component per element, 3 variant states: One line of JS on the root element switches all 3 color modes simultaneously. Zero duplicate code, zero maintenance overhead.

RETROSPECT

What I'd do differently.

Ship the mode switch, then build toward invisible infrastructure. The manual toggle works — a CVD user sets it once and forgets it. But the architecturally superior solution is OS-level detection: read prefers-color-scheme and forced-colors, then extend those signals to infer CVD preference at the system level.

I'd also run a second round of research specifically with mobile traders. The 3-mode system was designed and validated on desktop. The greyscale mode in particular has contrast assumptions that break down on OLED screens outdoors. That's a known gap I'd close before shipping v2.

TokenDrift — Catching Design Token Drift in Real-Time

A VS Code extension that detects hard-coded design values as you type, maps them to your token system, and prevents drift from ever reaching production.

TypeDeveloper Tool / Extension
StackTypeScript · VS Code API
Token FormatDTCG W3C Standard
SourceGitHub →

THE PROBLEM

Design tokens exist. Engineers ignore them. Nobody catches it until it's too late.

Every modern design system has a token library — a single source of truth for colors, spacing, and typography. The promise: change one token, update the entire product.

In practice, this breaks constantly. Under deadline pressure, engineers write #2563eb instead of var(--color-primary). They type 16px instead of var(--spacing-md). Each instance is a small divergence — but they compound. Within months, a codebase has hundreds of drift values, and the design system becomes increasingly irrelevant.

The problem wasn't carelessness. The problem was no feedback loop. Across 40 developers working on 12 products, this produces 500+ inconsistent design values — and a design system that's technically live but practically ignored.

THE DISCONNECT

The tokens existed. Nobody used them.

The token file was there. The CSS was there. But there was no link between them — no signal, no warning, no bridge. Engineers wrote what they knew.

HOW IT WORKS

Three layers of intelligence — one extension.

TokenDrift operates as a VS Code Language Server watching CSS, SCSS, TypeScript, and JSX files in real-time. It loads your tokens.dtcg.json and builds a reverse-lookup index — from raw value back to token name.

Yellow underline

Token match — quick fix available

The hard-coded value exactly matches an existing token. Hover to see which token it maps to. Press Ctrl+. to replace with var(--token-name) in one keystroke.

Blue underline

New token suggestion

The value doesn't match any token but appears above the configured threshold (default: 3×). TokenDrift suggests creating a new token and auto-generates the DTCG entry.

CI Gate

Pipeline enforcement at merge time

A GitHub Actions step that runs the same analysis on every PR. Configurable: fail above a drift score threshold, or post a comment summary.

Dashboard

Live team drift score

A lightweight vanilla HTML dashboard showing current drift score, most-violated tokens, file-level breakdown, and trend over time.

ARCHITECTURE

Built on the W3C standard. Designed for real engineering workflows.

TokenDrift uses the DTCG (Design Token Community Group) format — the W3C proposed standard for design token interoperability. It works natively with:

  • Figma Tokens — Export directly from Figma in DTCG format
  • Style Dictionary — The most widely used design token build tool
  • Theo (Salesforce) — Enterprise-scale token system
  • Custom token files — Any tokens.dtcg.json following the W3C spec

On large codebases with 500+ token definitions, analysis runs in under 200ms per file. Designed to never slow down the editor.

DESIGN DECISIONS

Why yellow and blue? And other decisions that matter.

  • Yellow for matches, not red: Red in VS Code implies broken code. A token match isn't broken — it's improvable. Yellow (warning) communicates the right severity.
  • Blue for suggestions, not yellow: Suggestions are speculative — "this might become a token." Blue in VS Code's diagnostic system means informational. Engineers already understand this convention.
  • DTCG format over custom JSON: Using the W3C standard means TokenDrift works with every existing toolchain — and will continue to as the ecosystem evolves.
  • Drift score, not just a lint count: A score (0–100) gives teams a benchmark to improve against, enables trending, and creates a shared definition of "good" across engineering and design.

RETROSPECT

What I'd do differently.

Talk to engineers before building, not after. I designed the threshold defaults (3× repetition before suggesting a new token) based on intuition and a small internal test. After shipping, I discovered the defaults were too aggressive — engineers were seeing blue suggestions on values they legitimately didn't intend to tokenize. The noise eroded trust in the extension fast.

The fix was a settings key (tokendrift.suggestionThreshold) which solved it technically — but if I'd run 5 engineer interviews before writing a line of TypeScript, I'd have calibrated it correctly from the start.
